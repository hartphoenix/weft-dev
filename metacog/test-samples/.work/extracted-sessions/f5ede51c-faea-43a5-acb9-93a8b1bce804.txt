# Plan: Tool-turn collapsing + prompt framing for MetaClaude observer

## Context

The meta-agent's 10-turn sliding window is dominated by low-information
tool-only entries (`[Bash]`, `[Read]`, etc.) — each ~6 chars. 66% of
assistant turns in a real session are tool-only. This means:

1. **Information starvation:** A 10-slot window during tool-heavy work
   may contain 1 text turn and 9 `[Bash]` entries (~91 chars total).
2. **False drift detection:** The meta-agent sees walls of `[Bash]` and
   concludes Builder Claude is off-task, when it's actually researching.

Simulation against transcript bc131056 (a 195-turn MetaClaude development
session) confirmed both problems. During turns 100-132, the meta-agent
fired 7 consecutive false-positive drift injections while Builder Claude
was exploring the codebase via Bash/Agent calls. Simulation logs:
`weft-dev/meta/simulation-log.jsonl` and `simulation-log-range.jsonl`.

With collapsing, a 10-slot window typically holds 6-8 substantive turns
(vs. 1-3 before). No window size increase needed — the density fix is
sufficient.

Two complementary fixes:
- **Structural:** Collapse consecutive tool-only turns into a single
  summary entry, so the 10-slot window holds 10 meaningful items.
- **Prompt framing:** Tell the meta-agent that tool sequences are
  expected research behavior, not drift signals.

### 1. observer.sh — tool-turn collapsing in jq pipeline

Insert a `reduce`-based transform between the initial array construction
and the `.[-10:]` slice. The transform:

- Detects tool-only entries via `test("^\\[.+\\]$")` (content is exactly
  `[SomeName]`)
- Collapses consecutive runs into a single `{role: "assistant", content:
  "[tools: Bash x4, Read x1, Agent x1]"}` entry
- Uses a transient `_tools` array as accumulator during the reduce pass,
  then formats and replaces in a second pass
- Non-tool entries and user entries pass through unchanged

**Replace lines 88-107** (the full RECENT assignment) with:

```bash
RECENT=$(tail -50 "$TRANSCRIPT_PATH" | jq -s '
    [.[] | select(.type == "user" or .type == "assistant") |
     {role: .type, content: (
         if .type == "user" then
             if (.message.content | type) == "string" then
                 .message.content | .[0:2000]
             else "" end
         else
             [(.message.content // [])[] |
              if .type == "text" then .text
              elif .type == "tool_use" then ("[" + .name + "]")
              else empty end
             ] | join("\n") | .[0:2000]
         end
     )} | select(.content | length > 0)
    ]
    | [ reduce .[] as $item (
        [];
        if ($item.content | test("^\\[.+\\]$"))
        then
          if (length > 0 and (.[-1] | has("_tools")))
          then .[-1]._tools += [$item.content[1:-1]]
          else . + [{"_tools": [$item.content[1:-1]]}]
          end
        else . + [$item]
        end
      ) | .[] |
      if has("_tools")
      then {"role":"assistant","content":(._tools | group_by(.) | map("\(.[0]) x\(length)") | join(", ") | "[tools: \(.)]")}
      else .
      end
    ]
    | .[-10:]
' 2>/dev/null || echo "[]")
```

**Note:** `group_by(.)` sorts tool names alphabetically. Output order
is deterministic (e.g., `Agent, Bash, Read`) regardless of call order.

**Why this is safe:**
- Output schema is identical: array of `{role, content}` objects
- `role` is always "user" or "assistant" — collapsed entries get
  "assistant", so `TURN_COUNT` (which counts `role == "user"`) is
  unaffected
- `content` is always a string
- Empty array input → empty array output (tested)
- No tool turns → passthrough (tested)
- The regex `^\\[.+\\]$` only matches entries that were already
  `[ToolName]` markers from the upstream extraction — it won't
  accidentally match text content containing brackets because the
  upstream extraction never produces text wrapped in `[...]`
  (The upstream jq only wraps content in `[...]` for `tool_use` blocks
  via `("[" + .name + "]")`. Text blocks are passed as raw `.text`.
  User content is passed as raw `.message.content`. No other code path
  produces bracket-wrapped strings.)

# Handoff Test

## The Naive Reader

A fresh Claude instance with CLAUDE.md and tooling but nothing from this conversation. The only things that exist for it are what's written in persistent artifacts.

## Process

1. Identify what was created or modified this session. Read each artifact in full.
2. From the naive reader perspective, find implicit dependencies:
   - Session-local terminology used without definition
   - References to "the bug," "the approach," "what we discussed" without grounding
   - Decisions recorded without rationale (a future reader needs *why* to know when the decision expires)
   - Scope assumed but not stated
   - Next steps that depend on conversation memory
