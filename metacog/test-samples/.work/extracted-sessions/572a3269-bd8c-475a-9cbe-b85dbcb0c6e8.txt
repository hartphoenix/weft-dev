[U4 18:26] Is it likely that using MLX models on LMStudio for inference model work contributed to the corruption of the GPU acceleration?
[A5 18:26] Plausible, yes. Here's the mechanism:

On Apple Silicon, the GPU is unified memory — the same physical RAM serves both CPU and GPU workloads. MLX specifically runs inference on the Metal GPU, and LM Studio with MLX models can consume significant GPU memory (a 7B model at Q4 is ~4GB of GPU-resident buffers, larger models proportionally more).

VS Code's WebGL terminal renderer maintains a **texture atlas** on the GPU — a bitmap of pre-rendered character glyphs. When GPU memory pressure increases:

1. The OS can evict or relocate GPU allocations
2. xterm.js's texture atlas can get corrupted or stale if the WebGL context loses sync with its backing memory
3. The "correct text underneath on highlight" symptom is consistent with the glyph cache being wrong while the text buffer (CPU-side) is fine

So the sequence would be: MLX loads fill GPU memory → WebGL texture atlas gets pressure → glyph cache corruption → visual artifacts accumulate over the session as the atlas gets progressively more out of sync.

This would also explain **why it only happens mid-session** — early in a session the atlas is fresh and small. Over time it grows, and the probability of a corruption event during GPU memory pressure increases.

**Way to test this hypothesis:** Run a Claude Code session of similar length *without* LM Studio running and see if the corruption onset is delayed or absent. If it is, you've isolated the contributing factor.

That said — the WebGL renderer has the texture corruption bug even without memory pressure. GPU contention from MLX would be an **accelerant**, not the root cause. Disabling `gpuAcceleration` would still be the robust fix regardless.
