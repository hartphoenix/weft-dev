# Sandboxing: From OS Primitives to Your Configuration Choices

## The Core Idea

A sandbox is a set of **kernel-enforced restrictions** on what a process can do. The key word is *kernel-enforced* ã the restrictions live below the process's privilege level, so the process cannot remove them by its own action. This is what distinguishes a sandbox from a convention, a hook, or a denylist.

Everything you know about iOS app sandboxing applies: each app gets a restricted view of the filesystem, can only access resources it has entitlements for, and can't escape those boundaries through its own code. The mechanism under the hood is the same on macOS ã it's just not mandatory for non-App-Store software.

## The Stack: How Isolation Actually Works

### Layer 1: Hardware + Kernel (the hard floor)

Every process on your machine already lives in a basic sandbox:

- **Virtual memory**: Each process gets its own address space. The CPU's MMU hardware enforces this ã Process A literally cannot address Process B's memory. This is a hardware guarantee.
- **Privilege rings**: Your code runs in Ring 3 (userspace). The kernel runs in Ring 0. Every interaction with the outside world ã files, network, other processes ã goes through a **system call** that transfers control to the kernel. The kernel is the gatekeeper.

Sandboxing = restricting what happens when a process makes syscalls. Everything else is implementation detail.

### Layer 2: System Call Filtering

**Linux (seccomp-BPF)**: A BPF program attached to a process that inspects every syscall and decides: allow, kill, return error, or trap. Once applied, it **cannot be removed** ã this is kernel-enforced. The filter sees syscall numbers and register values but deliberately cannot dereference pointers (prevents time-of-check/time-of-use attacks). Docker's default profile blocks ~44 syscalls.
