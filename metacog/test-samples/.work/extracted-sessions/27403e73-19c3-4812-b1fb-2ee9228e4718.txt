## Updated Analysis: Progress-Review Closes the P6 Loop

The previous analysis identified the gap: "Roger has no explicit mechanism for routing session findings back into system improvement." Progress-review is that mechanism. Here's what it actually does architecturally:

**The three-skill signal chain is now complete:**
```
session-review writes scores → startwork reads + dispatches progress-review →
progress-review reads across sessions → proposes state updates → human approves →
state files update → startwork reads updated state next session
```

This is the P6 embedding loop, actually closed. Not planned — built.

**Design decisions worth noting:**

1. **Confidence hierarchy across skills.** The scoring rubric formalizes: `session-review:quiz` (highest) > `session-review:observed` > `intake:artifact` > `intake:self-report` > `progress-review:pattern` (lowest). Pattern-inferred scores need ≥ 3 cross-session data points to override a session-review score. This resolves multi-source conflicts without ambiguity.

2. **Deferred findings that escalate.** Progress-review logs what it noticed but the user deferred. Next review, those findings get checked again — persistent patterns escalate in priority. The harness remembers what it noticed even when you postpone acting on it.

3. **Startwork as composition hub.** Startwork is no longer just a reader — it dispatches progress-review as a background sub-agent when enough sessions accumulate (>2 unreviewed). Its read-only contract is preserved for Phases 1-4; the writes happen in progress-review's scope after Phase 4 confirmation. This is a clean orchestration pattern.

4. **Dual activation.** The same phase structure serves both paths (dispatched from startwork, invoked standalone). The only difference is who gathers the data. Reusable pattern for any skill that should be both proactive and reactive.
