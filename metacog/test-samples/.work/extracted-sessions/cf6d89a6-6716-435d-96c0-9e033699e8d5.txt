# Lesson Scaffold

Restructure what someone else organized by procedure into what Hart needs organized by understanding.

## Inputs

1. **Source material** (required). URL, file path, or pasted text — any learning content.
2. **Current state** (optional). Energy, time, emotional context, what he needs. Shapes scope.

## Workflow

1. Fetch/read the source material.
2. Read `learning/current-state.md` for current learning state. Optionally read the most recent session log from `learning/session-logs/`.
3. Extract every concept the material teaches or assumes — explicit and implicit. Implicit concepts (things the lesson takes for granted) are often where the real learning edges are.
4. Cross-reference with current-state.md. What's solid, what's growing, what's new.
5. Produce a scaffold:
   - **Concepts to understand** — each with a one-sentence bridge to something Hart already knows. Flag current-state.md gaps by name.
   - **Execution sequence** — the instructor's steps reordered by conceptual dependency. Group steps that share a concept. Mark which concepts each step exercises.
   - **Resurfaced gaps** — low-scoring current-state.md concepts that this material touches. Spaced repetition for free.
6. Present and ask Hart whether the scope and classification feel right. He drives.

## Anti-Patterns

- **Don't execute the assignment.** Scaffold is for understanding, not walkthrough.
- **Don't teach the concepts yet.** Name and bridge them. Teaching happens during execution.
- **Don't skip implicit concepts.** Procedural lessons bury assumptions.
- **Don't ignore emotional context.** "I'm tired" is load-bearing information about session scope.

## Packaging (Docker)

**Image** — An immutable snapshot of a filesystem + installed software, ready to run. Same structure as a git commit: content-addressable, immutable, layered on top of previous state. You don't run an image; you instantiate it. Class → instance. Commit → working directory. Image → container. Same pattern at every scale.

**Container** — A running, isolated instance of an image. It has its own filesystem, its own network interface, its own process tree — but shares the host's kernel. The isolation boundary is the key concept: a container can't see other containers' files or ports unless you explicitly connect them. Same as function scope — local variables are invisible outside unless you return them.

**Dockerfile** — A declarative spec that produces an image. Sequence of instructions, each producing a **layer**. Layers are the critical concept: each instruction caches its result. If instruction 5 changes, layers 1–4 are reused, 5+ rebuild. Identical cache-invalidation logic as git rebase — amend commit 3, commits 4+ must replay. This is why Dockerfiles copy `package.json` before source code: dependencies change rarely, so that layer stays cached while your code layer rebuilds.

**Multi-stage build** — A Dockerfile with throwaway intermediate stages. Stage 1 installs dev dependencies and builds. Stage 2 copies only the build output into a clean image. Same as your Vite build: `node_modules` is 400MB of build tooling, `dist/` is 2MB of output. The production image only needs the 2MB.

**Volume** — A directory that persists outside the container's lifecycle. Containers are ephemeral — kill it, filesystem gone. A volume mounts a host directory into the container so data survives. Same as environment variables vs. hardcoded values: the volume externalizes state that shouldn't live inside the disposable unit.
