## Recommendation for session-review

The reliable retrieval pattern is:

1. `find ~/.claude/projects -maxdepth 2 -name "*.jsonl" -mtime -N | grep -v subagents` (where N covers the review window)

3. Filter by date range
4. Skip `<ide_opened_file>` and `<system-reminder>` lines when extracting first user prompt

The other two data sources (`sessions-index.json`, `history.jsonl`) are unreliable for current/recent sessions and shouldn't be used as primary lookups.

## What the script would do

A session-discovery script would:
1. Resolve the Claude data directory (default `~/.claude/`)
2. `find` JSONL files under `projects/` by modification date
3. Read first + last line of each match for timestamps
4. Output a JSON manifest: `{sessionId, project, start, end, msgCount, filePath, firstPrompt}`

That manifest is tiny — the agent receives a structured list instead of spending 5-15 tool calls discovering the same information. Analysis starts immediately.

## Resilience problem: `CLAUDE_CONFIG_DIR`

The two agents gave conflicting answers, and the truth is messy. There **is** an environment variable `CLAUDE_CONFIG_DIR` that can relocate the data directory, but:

- It's underdocumented and has unclear behavior ([GitHub #3833](https://github.com/anthropics/claude-code/issues/3833) — users report it creates hybrid state across both locations)
- It's not in `claude --help` output
- There's an open feature request ([#25762](https://github.com/anthropics/claude-code/issues/25762)) asking for better support
- Your machine doesn't have it set — you're on the default `~/.claude/`

So a hardcoded `~/.claude/projects/` path works for most users, but a robust script should check `CLAUDE_CONFIG_DIR` first:

```bash
CLAUDE_DIR="${CLAUDE_CONFIG_DIR:-$HOME/.claude}"
```

## Other resilience concerns

**Project path encoding** — The `/` → `-` transformation is undocumented and could change between Claude Code versions. But the script doesn't need to decode paths — it just needs to find JSONL files under `projects/`, which is a stable directory structure.

**JSONL schema** — The message format (`type`, `timestamp`, `message.content` structure) is internal and undocumented. It could change. The script should fail gracefully if it can't parse a file rather than crashing.

**30-day retention** — Claude Code says sessions are stored locally for up to 30 days. A script looking for sessions older than that may find nothing.

[A54 18:02] The version tells you whether the JSONL schema is the same one the script was written against. If Anthropic ships a Claude Code update that changes the message format — renames `type` to `role`, restructures `message.content`, changes the timestamp format — the script would silently return wrong data or empty results.

With the version, a consuming skill could check: "this session was recorded with v2.1.59 and I know how to parse v2.x — proceed" vs. "this says v3.0.0 and I've never seen that — fall back to agent-based discovery instead of trusting the parsed output."

**In practice right now:** probably not worth fixing. There's only one schema version in the wild, and if Anthropic changes it, the script will need updating regardless — the version field itself might move or disappear. It's a "nice to have if it's free" signal, not something to build logic around today.
