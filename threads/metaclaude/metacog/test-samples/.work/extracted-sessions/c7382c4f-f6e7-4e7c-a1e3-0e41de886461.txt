# Plan: MetaClaude Phase 2 Prerequisites + Artifact Migration

## Context

MetaClaude Local has Phase 1 largely complete and Phase 2 embedding index
built, but artifacts are scattered across three locations (weft-dev/meta/,
weft-dev/tools/, weft/tools/metaclaude/) with no experiment log despite
significant experiment data existing in JSONL logs, eval results, and the
PRD narrative. Two Phase 2 prerequisites (JSON format, configurable URL)
block clean Phase 2 wiring. API unification deferred. Several Phase 1
items (model comparison, prompt variants) are better done after the JSON
format switch and should resequence into early Phase 2.

**Goal:** Get organized, create the experiment log, fulfill Phase 2
prereqs, and position the project for replication and reporting.

### Boundary principle (weft/ vs weft-dev/)

- **weft/** — tools built for weft users (embedding pipeline, observer,
  skills, anything portable)
- **weft-dev/** — personal to Hart's use case (indexes, session logs,
  experiment data, benchmarks calibrated to his content)
- **roger/** — learning/growth model, not harness development

**Repos involved:** Steps 0-3 and 5 operate in `weft-dev` (this repo).
Step 4 operates in `weft` (sibling repo at `~/Documents/GitHub/weft`).
Each repo gets its own commit(s). Step 3's "single commit" applies to
weft-dev only. All changes to `weft/` must be committed to feature
branches off `hart/metaclaude` (not branched from or merged to main).

## Sequencing rationale

Phase 2 prereqs first, then migrate. The JSON format switch rewrites
observer.sh's parsing logic. Migrating files before that rewrite means
moving files then immediately rewriting them — wasted diff. The
experiment log is independent and can start immediately.

**Deferred:** API unification (routing --mch/--mco through Anthropic API
via curl instead of `claude -p`). Keep both code paths for now. The
payload composition divergence is acceptable — retrieval context can be
added to both paths when Phase 2 wiring happens. Revisit for Phase 4
model comparison if the two invocation mechanisms complicate benchmarking.

## Step 2: Switch XML tags to JSON response_format

**Problem:** `<inject>`/`<context>` XML parsing via perl regex is fragile.
Phase 2 adds retrieved chunks with source paths — more regex compounds
fragility. Phase 2.5 trains on the target format — switching after
training wastes the fine-tune.

**Files:**
- `weft/tools/metaclaude/prompt.md` — replace output format section:
  ```json
  {"inject": "observation text or null", "context": "updated session summary"}
  ```
  Use `null` for silence (replaces `[no comment]` sentinel — JSON has native null).
- `weft/tools/metaclaude/observer.sh` — replace perl regex extraction (the `# Parse structured response` section) with jq:
  ```bash
  INJECT_CONTENT=$(echo "$CLEAN_RESPONSE" | jq -r '.inject // empty')
  CONTEXT_CONTENT=$(echo "$CLEAN_RESPONSE" | jq -r '.context // empty')
  ```
  Add `response_format: {"type": "json_object"}` to curl payload.
- `weft-dev/meta/simulate-accumulator.sh` + `simulate-accumulator-range.sh` — same parsing changes, add `response_format` to curl.
- `weft/tools/metaclaude/test-parser.sh` — remove XML-specific tests (tag extraction, malformed XML, multi-byte boundary). Keep content-logic tests (silence detection via null check, accumulator preservation, tool-collapsing). Estimated test count drops from 29 to ~12-15.

**Risk:** Qwen3-8B JSON mode compliance. Mitigate: test 5+ turns manually before committing. If compliance is poor, keep XML fallback and revisit with a different model.

**Experiment data:** Log JSON compliance rate per model. Add entry to experiment log. Persist test transcript.
