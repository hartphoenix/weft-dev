[A2 21:21] ## Pre-Commit Secret Scanning: Ranked Candidates

### Rank 1: Gitleaks — best overall for your setup

`brew install gitleaks`. Go binary, fast, 160+ detectors (regex + entropy), scans staged files with `gitleaks protect --staged`. Active development (v8.30.0, Nov 2025). Inline `gitleaks:allow` comments and `.gitleaksignore` for false positives.

**Why it wins:** Fastest execution, simplest global setup, broadest detection for the weight. One binary, one command, zero runtime dependencies.

### Rank 2: TruffleHog — maximum detection if you accept the trade-offs

`brew install trufflehog`. 800+ detectors, and the unique feature: **credential verification** (confirms whether a detected secret is actually live by testing it against the service). Released v3.93.7 *today*. Extremely active.

**Trade-offs:** Heavier than gitleaks, makes network requests during commit (offline = secrets classified as "unknown"), slower execution. The verification feature nearly eliminates false positives for known secret types, which is genuinely powerful.

### Rank 3: Talisman — best global-first design, but install friction

Built by ThoughtWorks specifically as a global pre-commit/pre-push hook. Scans filenames + content + entropy. Checksum-based whitelisting (more secure than pattern-based).

**Trade-off:** Not in Homebrew — installs via curl-pipe-bash script. That friction matters for a tool whose value proposition is "always on everywhere."

### Skip these

- **git-secrets**: Last release 2019. AWS-only by default. Superseded.
- **detect-secrets**: Good baseline concept, but maintenance slowed (no release in 10 months), requires pip, per-repo baseline complicates global deployment.
- **secretlint**: Node.js dependency, global install has caveats, smaller community.

[A4 21:26] ## Fit Analysis Against the Settings Plan

### The critical interaction: sandbox + network

TruffleHog's distinguishing feature — credential verification — **makes network requests during the pre-commit hook**. When Claude runs `git commit` via Bash, the sandbox wraps the process tree. TruffleHog's verification calls would hit the sandbox proxy, causing either prompts or failures. This doesn't affect Hart running `git commit` manually (no sandbox), but it means TruffleHog's best feature is degraded or broken in the exact context where you most need the hook to work silently.

Gitleaks and Talisman make zero network calls during scanning. They work cleanly inside the sandbox.

### Global setup conflict

All three tools need `git config --global core.hooksPath` for global installation. That's **hard-denied** in the settings plan (`Bash(git config --global*)`). This is correct — Claude shouldn't touch it. Hart runs the one-time setup manually. No conflict, just noting it's a manual step.

### The `--no-verify` gap

None of the three tools can survive `git commit --no-verify`, which bypasses all client-side hooks. The settings plan **does not deny `--no-verify`**. A prompt injection could use:
```
git commit --no-verify -m "update"
```
...and skip the pre-commit scanner entirely. This is worth adding to the deny list:
```
"Bash(git commit --no-verify*)",
"Bash(git commit -n *)",
"Bash(git commit -n)"
```

### Talisman's install friction

Talisman installs via `curl | bash`. Under the new settings, shell piping works (bash chaining restriction lifted), and `curl` GET is allowed, but piping to `bash` is effectively `bash -c` territory — ask-gated. Again, Hart runs this manually so it's fine, but it's one more manual step vs. `brew install gitleaks`.
