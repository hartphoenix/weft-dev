[A48 01:37] The claimed sequence is wrong. Here's why.

## Claimed flow
> intake → session-review → lesson-scaffold → startwork (+ progress-review)

## Actual flow

The learning loop is a **cycle**, not a linear chain. Within a session, the order is:

```
intake (once, seeds everything)
  ↓
┌─────────── session cycle ───────────┐
│                                     │
│  startwork (session open)           │
│    ↓ dispatches progress-review     │
│    ↓ if unreviewed sessions > 2     │
│                                     │
│  lesson-scaffold (before work)      │
│    ↓ writes predictions to          │
│      scaffolds/ + session log stub  │
│                                     │
│  [ actual work ]                    │
│                                     │
│  session-review (session close)     │
│    ↓ validates scaffold predictions │
│    ↓ updates current-state, logs    │
│                                     │
└──────── feeds next cycle ───────────┘
```

**Three errors in the claimed sequence:**

1. **lesson-scaffold comes before session-review, not after.** Scaffold writes *predictions* about concept difficulty before the work happens. Session-review then *validates* those predictions against actual evidence. The scaffold → session-review direction is the calibration loop.

2. **startwork is the session opener, not the endpoint.** It reads all accumulated state to compose a briefing. It sits at the top of each cycle, not the end.

3. **The relationship between lesson-scaffold and session-review is bidirectional, not one-directional.** Session-review reads scaffold predictions to check for calibration mismatches (scaffold said "solid" but learner struggled). Lesson-scaffold reads session-log frontmatter to classify concepts. They feed each other across sessions.

**Progress-review** sits correctly as a side-channel — it's conditionally dispatched by startwork and reads across multiple sessions rather than within one.
