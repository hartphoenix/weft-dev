## How DAGs are commonly represented

The dominant pattern across every mature system I surveyed — package managers, build systems, curriculum mappers, knowledge graphs — is **distributed declaration, centralized resolution**. Each node declares its own relationships; the global graph is computed, not hand-maintained.

- **npm**: each `package.json` declares dependencies locally; `package-lock.json` is the resolved graph
- **Bazel**: each BUILD file declares its targets; the full DAG is computed at build time
- **Obsidian**: each markdown file contains wiki-links; the graph view is rendered from a scan
- **Airflow**: each Python file defines one DAG; metadata exports to JSON/YAML sidecars
- **ESCO/O*NET competency frameworks**: typed-edge directed graphs with multi-resolution views

Nobody stores a large graph as a single monolithic file in the canonical store. The graph is always either distributed across node-local declarations, or it's a computed/generated artifact derived from those declarations.

## JSON: can it represent DAGs fluently?

Short answer: **structurally yes, operationally no** — at least not for your use case. JSON can express adjacency lists and edge lists. But it fails on three constraints that matter here:

1. **Token cost.** JSON uses ~50% more tokens than YAML for equivalent data (braces, mandatory quotes, commas are pure overhead). A 200-node graph in JSON would consume ~28,000 tokens. YAML: ~18,000. Neither is great for a working agent's context window.

2. **No addressable sections.** An LLM can't be told "read the bridge dependencies section of this JSON" — JSON has no structural sections. You load it all or nothing.

3. **Incremental update.** Adding a node means modifying a nested object. Adding an edge means appending to an array inside the file. No append [...truncated]

The **linked markdown files are the canonical store**. That's where schema evolution happens — you add a field to frontmatter, change a field name, introduce a new relationship type.
